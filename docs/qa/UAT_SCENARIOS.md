# UC Check UAT 시나리오

UAT 목적: 실제 스태프가 Urban Conditioning 운영 상황을 가정해 UC Check가 업무 흐름을 방해하지 않고 정확히 동작하는지 확인한다.

테스트 기준일: 오늘 날짜, 한국 시간 기준.

---

## 시나리오 A — 태블릿 셀프 체크인 정상 흐름

대상 회원: 박서연 / 전화번호 끝 4자리 `1234` / 잔여 3회

1. 태블릿에서 `/kiosk/unlock` 접속
2. 직원 PIN 입력 후 `/kiosk` 이동
3. 전화번호 끝 4자리 `1234` 입력
4. 후보 목록에서 박OO 선택
5. 개인 PIN 입력
6. 출석 체크
7. 완료 화면 확인
8. 5초 후 `/kiosk` 복귀 확인

성공 기준:

- 후보 카드에는 마스킹된 정보만 표시된다.
- 출석 완료 후 잔여횟수가 2회로 표시된다.
- `/attendance/today`에 `source = kiosk`로 기록된다.
- 같은 회원이 다시 체크인하면 중복 출석이 막힌다.

---

## 시나리오 B — 동일 뒷자리 후보 선택

대상 회원: 박서연, 박성훈 / 둘 다 전화번호 끝 4자리 `1234`

1. `/kiosk`에서 `1234` 입력
2. 후보 카드가 2개 이상 표시되는지 확인
3. 각 카드에 전체 전화번호가 표시되지 않는지 확인
4. 잘못된 회원을 선택하고 뒤로가기
5. 올바른 회원 선택

성공 기준:

- 동일 뒷자리 회원이 있어도 앱이 깨지지 않는다.
- 메모, 결제 정보, 전체 연락처는 노출되지 않는다.
- 후보가 많을 때 스태프 문의 안내가 명확하다.

---

## 시나리오 C — 잔여 0회 회원 체크인 차단

대상 회원: 최유진 / 전화번호 끝 4자리 `9999` / 잔여 0회

1. `/kiosk`에서 `9999` 입력
2. 후보 선택
3. 확인 화면에서 출석 가능 여부 확인

성공 기준:

- 출석 체크 버튼이 비활성화되거나 스태프 문의 안내가 나온다.
- attendance_logs가 생성되지 않는다.
- remaining_sessions가 변경되지 않는다.

---

## 시나리오 D — 스태프 직접 출석 체크

대상 회원: 오민재 / 전화번호 끝 4자리 `8888` / 방문 이력 없음

1. owner/admin/coach 계정으로 로그인
2. `/check-in` 이동
3. `8888` 또는 회원명 검색
4. 활성 회원권 선택
5. 출석 체크
6. `/members/[id]`에서 last_visit_date 확인

성공 기준:

- 출석 로그가 생성된다.
- source는 `staff`다.
- 잔여횟수가 1회 차감된다.
- 회원 상세의 최근 방문일이 오늘로 바뀐다.

---

## 시나리오 E — 출석 취소

대상: 오늘 출석 처리된 회원 1명

1. admin 또는 owner로 로그인
2. `/attendance/today` 이동
3. 대상 출석 기록 선택
4. 취소 사유 입력
5. 출석 취소 실행

성공 기준:

- attendance_logs.status가 `cancelled`로 변경된다.
- cancelled_by, cancelled_at이 저장된다.
- 회원권 잔여횟수가 1회 복구된다.
- audit_logs에 ATTENDANCE_CANCELLED가 저장된다.
- coach/front_desk는 취소할 수 없다.

---

## 시나리오 F — 재등록 대상 관리

대상 회원: 박서연 / 잔여 1~3회

1. `/dashboard` 이동
2. 재등록 대상 패널 확인
3. 대상 회원 클릭
4. 회원 상세에서 잔여횟수와 리텐션 위험 점수 확인
5. 재등록 메모 작성

성공 기준:

- 잔여 1~3회 회원이 재등록 대상으로 표시된다.
- 메모 저장 후 타임라인에 표시된다.
- 키오스크 화면에는 메모가 표시되지 않는다.

---

## 시나리오 G — 장기 미방문 관리

대상 회원: 이정훈 / 14일 이상 미방문

1. `/dashboard` 이동
2. 장기 미방문 패널 확인
3. 대상 회원 상세 이동
4. 최근 4주 출석 패턴 확인
5. 상담 메모 작성

성공 기준:

- 장기 미방문 태그가 표시된다.
- 최근 4주 출석 패턴이 현재 출석 저하를 보여준다.
- 메모가 저장된다.

---

## 시나리오 H — 잔여횟수 조정

대상 회원: 임의 활성 회원권 보유 회원

1. admin 또는 owner로 로그인
2. 회원 상세 진입
3. 잔여횟수 조정 `+1` 선택
4. 사유 입력: `오입력 복구`
5. 저장
6. audit_logs 확인

성공 기준:

- remaining_sessions가 +1 된다.
- 사유 없이 저장되지 않는다.
- coach/front_desk는 실행할 수 없다.
- audit_logs에 PASS_REMAINING_ADJUSTED가 저장된다.

---

## 시나리오 I — 노쇼 기록

대상 회원: 이정훈 또는 임의 활성 회원

1. 회원 상세 진입
2. 활성 회원권 카드에서 노쇼 기록
3. 날짜 선택
4. 메모 입력
5. 저장
6. 같은 날짜로 다시 노쇼 기록 시도

성공 기준:

- attendance_logs.status = `no_show`
- deducted_sessions = 0
- remaining_sessions는 변하지 않는다.
- 같은 회원/회원권/날짜 중복 노쇼는 막힌다.
- audit_logs에 NO_SHOW_RECORDED가 저장된다.

---

## 시나리오 J — 비로그인 접근 보안

1. 브라우저 시크릿 창 열기
2. `/dashboard` 직접 접근
3. `/members/new` 직접 접근
4. `/settings/staff` 직접 접근
5. `/attendance/today` 직접 접근

성공 기준:

- 모든 staff 페이지는 로그인 없이 내용을 보여주지 않는다.
- 정적 HTML로 Demo Admin 또는 회원 데이터가 노출되지 않는다.
